# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A macOS menu-bar agent (`LSUIElement`, no Dock icon) that translates the current selection on **double Cmd+C** using a local LLM (Gemma via Ollama on `localhost:11434`) and streams the result into a floating panel near the cursor. Two empirical decisions are load-bearing and must not be changed lightly: `think:false` is mandatory (Gemma 4 otherwise burns 10–26s on reasoning tokens for an identical translation), and the native REST `/api/generate` is used over the OpenAI-compatible `/v1` layer (which doesn't expose `keep_alive`).

## Build & test

The Xcode project is **generated** by XcodeGen and is git-ignored. Never edit `*.xcodeproj` by hand — change `project.yml`, then regenerate. All four scripts regenerate the project first, so new source files are picked up automatically.

```bash
scripts/gen.sh      # regenerate Glosso.xcodeproj from project.yml
scripts/build.sh    # xcodegen generate + xcodebuild (Debug, arm64) → .build/dd/...
scripts/run.sh      # build + open the app (look for the bubble icon in the menu bar)
scripts/test.sh     # xcodebuild test → .build/TestResults.xcresult
scripts/package.sh  # Release build + signature-preserving zip → .build/release/Glosso.zip
```

`scripts/package.sh` produces a standalone `.zip` for a personal install: unzip, drag `Glosso.app` to `/Applications` (it then shows in Launchpad/Spotlight and runs without Xcode). It uses `ditto` (not `zip`) so the signature survives. Signed with the stable self-signed `Glosso Self-Signed` identity, **not notarized** (no Apple Developer account): first launch on another Mac needs a one-time right-click → **Open**. The "Launch at login" toggle (`SMAppService.mainApp`) only registers reliably once the app lives in `/Applications`.

**Builds and `xcodebuild` must run outside the sandbox** (`dangerouslyDisableSandbox: true`). Inside the sandbox they fail with `Operation not permitted`. `gh` and other network calls (GitHub API, Ollama over TLS) also need `dangerouslyDisableSandbox: true` — in the sandbox they fail with a TLS cert error (`OSStatus -26276`), not the build's `Operation not permitted`. During `/babysit-pr`, resolving a `claude-review` thread (`gh api graphql … resolveReviewThread`) is user-authorized in two cases: (1) findings actually fixed in committed code, and (2) findings you deliberately skipped after posting a rationale reply on the thread (low-value, or whose only fix would change a documented design decision). Leave a thread **unresolved only** when it is genuinely handed to a human — an un-adjudicable product/design call you did not decide yourself. The Bash tool's shell is **zsh** — unquoted `$var` is not word-split; iterate lists with `while IFS= read -r` (a `for x in $list` runs once on the whole string).

Signing uses a **stable self-signed certificate** (`Glosso Self-Signed`, `CODE_SIGN_STYLE: Manual`, no `DEVELOPMENT_TEAM` — created once in Keychain as Code Signing / Always Trust). It is deliberately constant because the Accessibility (TCC) grant is pinned to the signing identity: reusing the same cert keeps the grant across updates (verified A→B), whereas an ad-hoc signature would reset it on every release. Hardened runtime is **Release-only** — Debug sets `ENABLE_HARDENED_RUNTIME: NO` and `ENABLE_DEBUG_DYLIB: NO`, otherwise library validation rejects the self-signed (no-Team-ID) unit-test bundle and `scripts/test.sh` crashes at bootstrap with "different Team IDs". `CFBundleShortVersionString` is bound to `$(MARKETING_VERSION)`. Switching the signing identity needs a one-time re-grant (`tccutil reset Accessibility com.mirek.glosso`, then re-add the app). Releases auto-ship via `.github/workflows/release.yml` on **merge to `main`**: a cheap check job compares `MARKETING_VERSION` against published releases and only then runs the macOS build → sign from CI secrets → `gh release create` (which makes the `vX.Y.Z` tag). Bumping the version in a PR is the release trigger — no manual tagging; merges that don't bump it are no-ops. The same job `sed`-repoints the hardcoded `releases/download/vX.Y.Z/Glosso.zip` links in `docs/index.html` to the new tag and commits it, so those pinned-version links are intentional and auto-maintained — don't "fix" them by hand. The repo is public so the update check and asset download work unauthenticated — see `docs/DISTRIBUTION.md`.

Live "Cannot find type X" / "No such module 'Testing'" diagnostics in the editor before a build are **SourceKit noise** (single module, not yet indexed), not real errors — verify against an actual `scripts/build.sh`/`test.sh` run.

Tooling **cannot screenshot the menu-bar popup or Settings window** — macOS TCC blocks capturing native windows, and `agent-browser` only drives web/Electron. Verify UI changes by `scripts/run.sh` (builds + launches) and asking the user for a screenshot. The **`docs/` landing is web**, though, so it *is* verifiable headlessly: serve it (`python3 -m http.server`, needs `dangerouslyDisableSandbox`) and drive agent-browser (`set viewport W H`, `screenshot --full`).

Tests use the **Swift Testing** framework (`import Testing`, `@Test`, `@Suite`, `#expect`) — not XCTest. To run one suite or test, add `-only-testing`:

```bash
xcodebuild test -project Glosso.xcodeproj -scheme Glosso \
  -destination 'platform=macOS,arch=arm64' -derivedDataPath .build/dd \
  -only-testing:GlossoTests/AppCoordinatorTests
```

`scripts/test.sh` prints only a terse "done" — for pass/fail/skip counts run `xcrun xcresulttool get test-results summary --path .build/TestResults.xcresult`. To iterate on a `PromptBuilder` change against the real model without rebuilding the app, POST the built prompt straight to Ollama: `curl -s localhost:11434/api/generate -d '{"model":"gemma4:26b-mlx","prompt":"…","stream":false,"think":false,"options":{"temperature":0},"keep_alive":"30m"}' | jq -r .response` (needs `dangerouslyDisableSandbox`). Mirror the locked options so the output matches the app's.

## Architecture

**`Sources/Core/Contracts.swift` is the frozen seam.** Every module talks to the others only through the protocols and value types defined there (`LLMClient`, `HotkeyMonitor`, `PasteboardReading`, `AXSelectionReading`, `TranslationPopupPresenting`, `AccessibilityAuthorizing`, `DoubleKeyDetecting`, `ModelListing`, `EngineProviding`, `ModelManaging`, `LoginItemManaging`, `SelectionReplacing`, plus the `TranslationEvent`/`TranslationError`/`CaptureError`/`SecondLanguage`/`Formality` enums, `Action`, `KeyChord` and `LLMConfig`). `LLMClient.run(_:action:model:second:formality:humanize:style:)` and `prewarm(model:)` take the user-selected model, second language and palette verb (`Action` — issue #23) per call; `humanize` is a translate-only modifier, `style` its `fixGrammar`-only twin (moderate style pass, issue #73). The `Action` **case order is load-bearing**: it drives both the popup's verb-pill order and the background prefetch priority. `alternatives(for:in:source:second:model:)` (non-streaming, returns `[String]`) and `reword(original:to:in:source:second:formality:model:)` (streaming, like `run`) back the per-word alternatives feature (issue #17); `explain(word:in:source:second:model:)` backs the per-word "Dlaczego tak?" line (issue #39), `explainFix(error:correction:original:corrected:second:englishRules:style:model:)` the grammar-diff fix reasons (issue #51), and `reply(to:model:)` returns several reply drafts for the `reply` verb (issue #60), which rides the same pipeline but takes this non-streaming list path, not `run()`. Concrete types are injected; tests swap in fakes from `Tests/Support/`. When changing a cross-module behavior, change the protocol here first.

`AppCoordinator` (`Sources/Coordinator/`, `@MainActor`) wires the modules and owns the flow. `AppDelegate` (`Sources/App/GlossoApp.swift`) builds the real instances and starts it; it **early-returns under XCTest** (`XCTestConfigurationFilePath`) so the app side effects don't fire during tests. The coordinator also owns the **action cache + prefetch** (issue #58/PR #77): finished results are cached per `Action` so switching verbs is instant, a single background `prefetchTask` pre-generates the other verbs in `Action.allCases` order, and the cache is dropped when its signature (model/second/humanize snapshot) changes, on reword undo, and on popup dismiss (which also clears `lastCapture` so a per-word fetch can't resurrect a stale prefetch).

The functional modules, each behind a protocol:

- **Hotkey** (`Sources/Hotkey/`) — `GlobalHotkeyMonitor` is a *passive* `NSEvent.addGlobalMonitorForEvents(.keyDown)` watcher matching the **typed character** "c" + Command (not the physical keyCode 8, so it also fires on Dvorak/AZERTY/Colemak). Both Cmd+C presses still copy normally. `DoubleCopyDetector` decides "double" by measuring the gap (~0.3s window) and resets after a hit so a third rapid press can't pair with the second. This is why the only OS permission needed is **Accessibility**, not a registered hotkey. The same monitor also resolves two **configurable headless chords** (`ChordHit.fix`/`.translate`, issues #21/#46): "fix grammar in place" (default Ctrl+Cmd+G) and "translate in place" (default Ctrl+Cmd+T), read live from `SettingsStore` via `chordProvider` and matched by character (`resolveChord` is pure/testable), firing `onFixGrammarInPlace`/`onTranslateInPlace`.
- **Capture** (`Sources/Capture/`) — reads `NSPasteboard`. `SelectionGuard` enforces that `changeCount` rose above a baseline (proof the user actually copied) and that the text is non-empty. When the poll loop times out (the app never copied on Cmd+C — Safari/WebKit does this inconsistently), `AppCoordinator` falls back to `AXSelectionReader` (`AXSelectionReading`), which reads the focused element's `AXSelectedText` directly via the Accessibility API — no pasteboard, no new permission (the AX consent the hotkey monitor already needs covers it). Because that read resolves whatever is focused *at fallback time* (~480ms after the press), `AppCoordinator` snapshots the frontmost app's PID at the double-press and bails out of the AX fallback if it changed (a Cmd+Tab mid-poll), so it never translates another app's selection. Deeper fallbacks (AppleScript, synthesized Cmd+C, Safari-via-JavaScript) stay deferred.
- **LLM** (`Sources/LLM/`) — `OllamaClient` POSTs to `/api/generate` with `stream:true`, parses the NDJSON line stream (`NDJSONStreamParser`), and yields `.token`/`.finished` through an `AsyncThrowingStream`. `PromptBuilder.build(for:action:second:formality:humanize:style:)` carries the **PL↔(selected second language) swap logic inside the prompt itself** and branches per palette verb (`Action` — issue #23): `translate` (the default, with the optional default-on humanizer directive), `summarize`, `fixGrammar` (with the optional `style` pass) and `reply` (drafts list, parsed by `ReplyParser` — issue #60). Plus the per-word `buildAlternatives` (one-per-line list, parsed by `AlternativesParser`) and `buildReword` (constrained re-translation) prompts for issue #17. Fix reasons are **grounded in static rule bases** (issues #73/#76): `PolishSpellingRules` (RJP-derived spelling + style cards) and `EnglishGrammarRules` (typical Polish-speaker errors); `englishRules` picks the grounding from the detected *source* language of the correction run — detection can misfire on short/mixed text, so the flag is passed in from the run, not re-detected per tap — and the Polish style cards join the base only when the run had `style` on. `prewarm(model:)` is best-effort (failures are swallowed) and keeps the model resident via `keep_alive`. Any directive folded into the translate prompt must stay **subordinate to the translation and name the target language**: a default-on humanizer phrased in English-style terms ("write as natural prose, avoid em dashes…") made Gemma rewrite an English source *in English* instead of translating it to Polish — verify both directions against live Ollama after any prompt-wording change.
- **Popup** (`Sources/Popup/`) — `TranslationPopupController` shows a borderless non-activating `FloatingPanel` that never becomes key (won't steal focus). Esc dismisses it (via the CGEventTap below) and the title-bar close button closes it; a click *outside* the window closes only an open alternatives dropdown (issue #30), never the panel — that outside-click-to-dismiss-the-panel was deliberately dropped in `5f4a549` once the window became movable with a close button. **The controller owns the window size** — the only stable arrangement found (the resizable-popup work): the hosting view runs with `sizingOptions = []` (no constraints at all), PopupView measures its own ideal size in SwiftUI (`.fixedSize()` + `.onGeometryChange` — fixedSize is load-bearing: under a concrete proposal the panes' ScrollViews greedily fill the window instead of reporting content size) and reports it to `applyContentSize()`, which — coalesced to the next runloop turn — sets the whole frame with the top-left pinned so the panel grows downward, not up over the cursor. Every alternative was tried and fails: `.preferredContentSize` resizes the window from inside window layout (`updateAnimatedWindowSize`) and recurses to a stack overflow under a live grip drag; `.intrinsicContentSize` installs real constraints and the window's autolayout repositions the window with a drifting origin; competing `setFrame` calls get reverted synchronously; pinning the SwiftUI root frame recurses layout. The text panes cap height at 400pt and scroll; the bottom-right grip (`ResizeGripArea`, an `NSView` — a SwiftUI `DragGesture` never receives the drag) stretches pane widths and that cap via `PopupModel.sizeDelta`, and the window follows the grown content through the same single path. **Window moving** is a `WindowDragGesture` + `.allowsWindowActivationEvents()` on the card (interactive children win arbitration; it cannot fire over the grip — events over an `NSViewRepresentable` never enter SwiftUI's gesture graph), with `isMovableByWindowBackground` permanently **false**: the background-drag region is published to the WindowServer ahead of time for the whole window, so with it enabled a grip drag also moved the window and nothing reactive (hover toggles, dynamic getters, dummy gestures) can stop it. Shadows are drawn in SwiftUI, not by AppKit (`FloatingPanel.hasShadow` is off): the system window shadow rings the dropdown's protruding part with a hard contour, so the panel and the dropdown each carry their own `.shadow()`. The window reserves a transparent `PopupView.shadowMargin` around the visible panel for the blur, and `present()` shifts the window top-left up-left by that margin to keep the panel under the cursor — any positioning change must account for it. In the `.done` phase the result becomes a `FlowLayout` of individually clickable words (`WordFlow.swift`: `Tokenizer`, `FlowLayout`, `AlternativesDropdown`); clicking one opens an alternatives dropdown rendered as an **in-window SwiftUI overlay** (`overlayPreferenceValue` anchored to the word) — deliberately **not** an `NSPopover`/`Menu`, which would force the panel to become key and break the non-activating focus model. Esc is layered: it closes an open dropdown first, then dismisses the panel. This trades away drag-to-select on the result (the header Copy button still copies the whole translation). The source pane is an editable `TextField` (`sourceText`, issue #44) — edits plus an explicit re-translate re-run over the changed text. In the `fixGrammar` action the `.done` phase renders a **grammar diff** (`GrammarDiff.swift`) instead of the word flow: changed spans are tappable and open the dropdown straight into a one-line Polish "why" reason (`fixReasonMode`, issue #51), which is why Esc layering special-cases it (no list to peel back to). A dense diff (>3 changes, `PopupModel.splitFixView`) splits the pane into a diff section plus a clean corrected-text section with an eye toggle that hides the diff (PR #79) — `diffHidden` is deliberately transient (reset on every fresh capture) so the learning layer can't stay silently off, and hiding closes any open reason dropdown since its anchor vanishes. The `reply` verb renders its drafts as a pick-one card list instead (issue #60).

Three smaller supporting modules: **Engine** (`Sources/Engine/`) — `EngineManager` (`EngineProviding`) finds or provisions the Ollama engine (`ensureEngine`, idempotent) via `OllamaEngineDownloader`, and `OllamaModelManager` (`ModelManaging`) pulls/deletes models through `/api/pull` (`PullProgressParser`). **Replace** (`Sources/Replace/`) — `SystemSelectionReplacer` (`SelectionReplacing`) pastes via a synthesized Cmd+V on `.cgAnnotatedSessionEventTap` with clipboard save/restore and a suppression filter — deliberately **not** AX-based replacement, which reports false success in web/Electron (issue #22). **Notify** (`Sources/Notify/`) — `SystemUserNotifier` posts user notifications for the headless in-place paths (success/failure/clipboard-fallback messages) and the new-version alert.

A **Settings** module (`Sources/Settings/`) holds the user-editable config: `SettingsStore` (`@Observable`, UserDefaults-backed) persists the Ollama `model`, the `secondLanguage` (the non-Polish side of the pair; English default), the popup toggles (`formality`, `humanize`, `fixStyle`), `lastNotifiedVersion` for the update alert, and the two headless chords (`fixChord`/`translateInPlaceChord`, `KeyChord` JSON-encoded; defaults in `Contracts.swift`), the latter edited via the click-to-record `KeyChordRecorder`. `SettingsView` is a SwiftUI `Settings` scene opened via `SettingsLink` in the menu; the model section is a unified radio list of the curated `EmbeddedModelCatalog` tiers (a gauge icon per speed↔accuracy tier with the friendly name in the tooltip/`accessibilityLabel`, per-row download/delete via `ModelManaging`, `Usuń` disabled on the active model), with the tier best-fitting this Mac's RAM (`EmbeddedModelCatalog.recommended`) badged "Zalecany"; downloading a model auto-activates it. Non-catalog installed models collapse into one right-aligned picker. `OllamaModelLister` (`ModelListing`, `GET /api/tags`, derived from the generate endpoint's host) still supplies the live installed-model list. There is no longer an engine row — a model download (`startPull`) provisions the engine lazily by calling `ensureEngine` (idempotent — a no-op when an engine is reachable) first. A "Launch at login" toggle binds to `SettingsStore.launchAtLogin`, which is **not** UserDefaults-backed: it derives from and drives `SMAppService.mainApp` through `LoginItemManaging` (`SMAppServiceLoginItem`), so the actual system registration is the source of truth (a user revoking it in System Settings is reflected on `refreshLaunchAtLogin`, called when the window appears). The empirical invariants (`think:false`, `temperature:0`, `keep_alive`, `endpoint`) stay locked in `OllamaClient`'s base config and are **not** exposed in the UI.

A first-run **wizard** (`OnboardingView`/`OnboardingController`, gated by `SettingsStore.hasCompletedOnboarding`) walks the user through model download → second language → usage + the one-time Accessibility grant; its model-pull flow is shared with Settings via `downloadModel()` (`ModelDownload.swift`). A best-effort **update check** (`Sources/Update/UpdateChecker.swift`) polls the public GitHub `releases/latest` (numeric version compare so `1.10 > 1.9`, silent on any failure), surfaces a menu-bar line plus a one-per-version notification (`lastNotifiedVersion`), and on click `UpdateDownloader` fetches the `.zip` into ~/Downloads and reveals it in Finder — a URLSession download carries no quarantine flag, so this skips Gatekeeper's "Open Anyway" dance; installing stays a manual drag-over (keeps the TCC grant) — deliberately no Sparkle/appcast.

### Three subtleties that aren't obvious from a single file

1. **The capture poll loop** (`AppCoordinator.captureAndTranslate`): the second Cmd+C only *triggers* the copy, so at the instant the double-press is detected the new clipboard contents aren't there yet. The monitor samples the baseline `changeCount` at the *first* Cmd+C (`registerPress`) — so it precedes the second copy even for apps that copy synchronously on keyDown (Chromium/Electron, Java) — and hands it to the coordinator via `onDoubleCopy`, which polls `readSelection` until `changeCount` rises (default 12ms × 40 attempts ≈ 480ms, to tolerate slow apps' copies) before streaming.

2. **`DirectionDetector` decides the translation direction.** `detect(_:second:)` uses `NLLanguageRecognizer` (constrained to Polish + the selected second language) to pick both the arrow label (PL→XX / XX→PL) **and** the target language named in the translate prompt: `PromptBuilder.instruction(for:second:formality:)` emits an unconditional "Translate … into X." Detection below a 0.8 confidence floor returns `.unknown` — short PL/EN homographs ("To", "Do") misdetect at ≤0.75 and, with the target now code-chosen, would translate English into English; correct reads measure ≥0.84. The earlier conditional swap in the prompt ("If it is Polish → X; otherwise → Polish") is kept only as the `.unknown` fallback — asking Gemma (`think:false`) to classify *and* translate in one step failed for non-English pairs (NL/RU sources came back echoed or lightly paraphrased instead of translated to Polish; verified live both with and without the humanizer).

3. **The headless in-place path** (`AppCoordinator.fixGrammarInPlace`, issues #46/#21): unlike double-Cmd+C, the chord triggers no copy, so it reads the selection straight from the focused element's `AXSelectedText` (falling back to a synthesized Cmd+C with clipboard save/restore when AX reads empty, e.g. web/Electron), runs `fixGrammar` or `translate` silently into a buffer, and pastes the result back — no popup. Like `handleReplace`, it snapshots the source app's PID up front and bails (handing the result to the clipboard instead) if focus moved, so it never pastes into the wrong app.

### Config & permissions

`LLMConfig.default` (model `gemma4:26b-mlx`, `think:false`, `temperature:0`, `keepAlive:"30m"`) lives in `Contracts.swift`; its invariants stay canonical, but the fresh-install **model** is now seeded from `EmbeddedModelCatalog.recommended` (the catalog tier that fits this Mac's RAM via `ProcessInfo.physicalMemory`), not `LLMConfig.default.model`. The catalog is 3 desktop Gemma tiers (`e4b` is mobile-only and excluded). At runtime the `model` and the second language come from `SettingsStore` (UserDefaults) per translation; `think`/`temperature`/`keepAlive`/`endpoint` stay fixed in `OllamaClient`'s base config. The app is **not sandboxed** (`Generated/Glosso.entitlements` sets `app-sandbox: false`, `network.client: true`) because Accessibility is incompatible with MAS sandboxing. Deployment target is macOS 26.0, Swift 6 (strict concurrency — note the `MainActor.assumeIsolated` shims around non-isolated AppKit monitor callbacks).

App icon and the custom menu-bar glyph live in `Sources/Assets.xcassets/` (`AppIcon.appiconset`, `MenuBarIcon.imageset`); the icon name is wired via `ASSETCATALOG_COMPILER_APPICON_NAME: AppIcon` in `project.yml`.

## Tests

- `Tests/` — fast unit tests, no network, using the fakes in `Tests/Support/`.
- `TestsIntegration/OllamaLiveTests.swift` — hits a real Ollama and **silently skips (returns early) when `localhost:11434` is unreachable**, so the suite stays green without a running daemon.
- Every test/helper file needs `@testable import Glosso` to see our types, and `import Foundation` when it uses Foundation types (e.g. `TimeInterval`). Helpers under `Tests/Support/` are easy to forget.
